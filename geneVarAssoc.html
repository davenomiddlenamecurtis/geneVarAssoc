geneVarAssoc

geneVarAssoc is a utility to set up analyses with scoreassoc, which is documented separately. scoreassoc is available at: github.com/davenomiddlenamecurtis/scoreassoc

There is no formal published account of geneVarAssoc but the actions it performs were described in this publication, which could be cited:

Practical experience of the application of a weighted burden est to whole exome sequence data for obesity and schizophrenia. D Curtis and the UK10K Consortium. Ann Hum Genet 80, 38-49. 2016.

Broadly, geneVarAssoc takes vcf files for cases and controls, uses tabix to extract variants for a gene, produces the necessary input files for scoreassoc and then runs scoreassoc.

Git

github.com/davenomiddlenamecurtis/geneVarAssoc – latest version of code

Compilation

Change to the src folder and then run:

make –f geneVarAssoc.mak

This command should produce executables called geneVarAssoc and intVarAssoc, which should then be copied to the directory assigned to DCBIN in geneVarAssoc.mak, by default called ../bin.

Solution and project files are provided to allow the program to be compiled with Microsoft Visual Studio. To compile for MSDOS/Windows, the preprocessor variable MSDOS must be defined.

Running geneVarAssoc

Typical usage:

geneVarAssoc --arg-file ../pars/gva.SSS.ct08.rare.arg --gene DRD2

The arguments for the program can be specified on the command line or in a file or using a combination of both. The arguments are in pairs, with the first member of the pair, starting with --, specifying the option to be set and the second member the value for that option.

When vcf file names are specified, if they contain the * character then this means that there are different vcf files for each chromosome and the * will be substituted with the chromosome number to extract the relevant variants for a particular gene.

For the simplest usage, there will be one or more vcf files containing control genotypes and one or more vcf files containing case genotypes. However one may also work with only allele frequencies, rather than genotypes, for controls and/or cases. If the control and case genotypes are in the same vcf file then the phenotypes are specified with a  separate file. If the subjects consist of trios then a separate file is used to define the pedigree relationships.

In order to run geneVarAssoc you will need the following:

A set of bgzipped vcf files which have been indexed with tabix and accompanied by their vcf.gz.tbi files. Case and control files can be separate and there can be up to ten case and ten control files. If separate case and control files are not used then one or more combined files can hold all the subjects, in which case one or more separate files specifying phenotypes must be provided. If the subjects consist of sets of trios then no phenotypes need be provided and it is assumed that the children are all affected. Generally the vcf files will contain genotypes but they can instead contain genotype probabilities as output by IMPUTE2. If individual level data is not available then instead a file of allele frequencies for cases or controls or both can be provided, e.g. from 1KG or ExAC. If this is done, the number of subjects used to produce these frequencies must also be given. There can be a separate vcf file for each chromosome or else all chromosomes can be combined.

 

There must be a file containing the coordinates of transcripts in refseq format and sorted by gene so that all the transcripts for each gene are grouped together, as in the provided example file refseqgenesorted101115.txt.

 

If the inbuilt annotation system is to be used, the human reference sequence must be provided in files named chr1.fa, chr2.fa, etc.

 

The programs tabix and scoreassoc must both be on the PATH.

When geneVarAssoc is run it uses tabix to extract the subset of variants matching the gene coordinates. Then it annotates them. Then it produces scoreassoc data files for valid variants and produces weights according to the annotations. Then it runs scoreassoc.

Typically a parameter file specified by --arg-file would contain the specifications for the analyses and the locations of the data and reference files and then the --gene argument would specify the gene to be analysed. Two example parameter files are provided. In exampleContFreqFile.arg, a 1KG file is used to provide allele frequencies based on 195 subjects and the case genotypes would be in files called BP_chr1.vcf.gz, BP_chr2.vcf.gz, etc. In gva.SSS.ct08.rare.arg, all the case and control genotypes are in scz-only-exome-all-nozeros.vcf.gz and the phenotypes are specified in SSSPhenotypes.txt. A number of other example files are provided which show how the different arguments and expressions can be used.

Because of the identifiable nature of sequence data, it is not possible to provide example data files.

When trying to get the program running, one can inspect the temporary files which are produced in order to elucidate problems. The most frequent reason for tabix to fail to extract any variants is a mismatch between the --add-chr variable and the way variants are named in the vcf file. That is, if the vcf file refers to a chromosome as e.g. chr10 then one must use --add-chr 1.

All the arguments are in pairs. The first member of the pair must begin with a double-hyphen and the second member provides the value for the argument, which for an option to be turned on or off is 1 or 0. The arguments can be provided on the command line or in argument files or in a combination of both. Arguments are separated by any amount of white space. If an argument value includes spaces then it must be enclosed in single quotes. For arguments on the command line, the operating system will apply the normal command line substitutions, which may affect arguments including asterisks and/or double quotes, so these should either be within an argument file (probably easiest) or else enclosed in single quotes to prevent substitution.

Here is the list of options:

--arg-file filename

Specifies the name of a file from which to read additional arguments. Once this file has been read successfully then further options will be read from the command line. One argument file can include another one using this option so that argument files can be nested and/or chained. The simplest way to run the program is to put all the arguments in a single file (e.g. options.arg) and then to enter:

geneVarAssoc --arg-file options.arg --gene DRD2

--ref-gene-file filename

Specifies a file in refseq format, with first line beginning:

#bin    name    chrom   strand  txStart . . . 

The file must have been sorted on the gene name to bring all transcripts for each gene together. E.g. refseqgenesorted101115.txt, which is provided in the exampleFiles directory.

--reference-path pathname

Provides the name of the folder containing the reference genome files, required if the inbuilt annotation functions are used, having names such as chr17.fa, etc.

--cont-file filename

Provides the name of a vcf file containing control subject genotypes. There can be more than one of these.

--case-file filename

Provides the name of a vcf file containing case subject genotypes. There can be more than one of these. If case and control subjects are in the same file or files then these should be specified as case files and the phenotype status should be specified using the --phenotype-file option. If trios are used then it is assumed that the child in each trio is affected and the vcf file containing all subjects is specified using --case-file while the trio structure is defined using --trio-file.

--phenotype-file filename

Specifies the name of a file which provides phenotypes of the subjects in the vcf file(s). There is one row per subject, consisting of a 0 or 1 (for case or control) or any other number (to omit this subject from the analysis). The rows must exactly match the order of subjects in the vcf file(s). This option can be used if one does not have separate vcf files for cases and controls. Typically one would use it with just a single vcf file. However, if for some reason one does wish to use this option with multiple vcf files please note that any control files will be read in before any case files, regardless of which are specified first. So the phenotype file must have phenotypes for the subjects in the control files before those in the case files. This aside, the vcf files are read in in the order in which they are specified. This argument can be repeated if there is more than one file containing phenotypes. The phenotypes in this file or files will override the control or case file specification. (A possible application of this would be if one wished to perform some kind of permutation testing.)

--ID-and-phenotype-file filename

Specifies the name of a file which provides phenotypes of the subjects in the vcf file(s). There is one row per subject, consisting of the subject ID as in the vcf file followed by a space then a 0 or 1 (for case or control) or any other number (to be ignored). The order does not need to match the order in the vcf file(s). If a subject has phenotype other than 0 or 1 or if there is no entry for a subject then it will not be included in analysis. This argument can be repeated if there is more than one file containing phenotypes.

--samples-file filename

Specifies the name of a file which provides phenotypes in the format used by IMPUTE2. The program looks for a column whose name begins PH (case sensitive) and then reads phenotypes (0 or 1) from that column. The order must match that of the subjects in the vcf file(s), as described for the --phenotype-file option. This argument can be repeated if there is more than one file containing phenotypes.

--use-probs 0 or 1

A value of 1 specifies that the vcf file contains genotype probabilities as output by IMPUTE2 rather than genotypes (denoted by 0).

--gene name

Provide the name of the gene to be extracted and analysed. Must match the gene name in the refseq file and the match is case-sensitive.

--trio-file filename

Indicates that the subjects consist of sets of trios as defined in the file specified. Each subject to be used occupies one row. The first entry is the ID of the subject, as in the vcf file. If the subject has parents then their IDs are given next, father then mother. If the subject is a parent then the parental IDs are given as 0 0. Subjects in the vcf file but not in this file will be ignored. Subjects with parental IDs 0 0 but not specified as parents by any other subject will be ignored.

--cont-freq-file filename

Specifies a vcf file providing the allele frequencies of variants in controls. If this option is used there will be no vcf file for control genotypes.

--num-cont N

Specifies the number of controls which had been used to obtain the control allele frequencies, i.e. the total number of controls genotyped. The number of successful genotypes may vary slightly between variants but the maximum number should be given here.

--case-freq-file filename

Specifies a vcf file providing the allele frequencies of variants in cases. If this option is used there will be no vcf file for case genotypes.

--num-case N

Specifies the number of cases used to obtain the case allele frequencies, i.e. the total number of cases genotyped. The number of successful genotypes may vary slightly between variants but the maximum number should be given here.

--allele-freq-str string

String used to recognise the allele frequency entry in a vcf file which is used to provide allele frequencies for cases or controls. If the vcf file contains entries such as …;EUR_AF=0.42;… then one would specify --allele-freq-str EUR_AF

--allele-number-str string

String used to recognise the allele number entry in a vcf file which is used to provide allele frequencies for cases or controls. If the vcf file contains entries such as …;EUR_AN=2414;… then one would specify --allele-number-str EUR_AN

--allele-count-str string

String used to recognise the alternate allele count entry in a vcf file which is used to provide allele frequencies for cases or controls. If the vcf file contains entries such as …;EUR_AC=27;… then one would specify --allele-number-str EUR_AC. If there is no allele frequency field then the frequency will be calculated as the count divided by the number (as in ExAC).

--bait-file filename

File containing coordinates for the baits. If this is specified then they will be used to determine the coordinates of exomes to extract rather than relying only on the gene coordinates in the refseq file.

--upstream number

Number of bases upstream of the gene to extract.

--downstream number

Number of bases downstream of the gene to extract.

--margin number

Number of additional bases around the baits to extract.

--weight-factor wf

Weight factor to provide to scoreassoc for weighting based on allele frequency, meaning that rarer variants receive a higher weight. Default value is 10, meaning very rare variants have 10 times the weight of very common ones. See scoreassoc documentation for further details.

--use-ensembl 0 or 1

If 1, use the variant effect predictor script provided by  ensembl to carry out annotation, otherwise use geneVarAssoc's own inbuilt annotation routines. In order to use this, you must have the perl version of variant effect predictor (VEP) set up so that commands such as the following can run:

perl variant_effect_predictor.pl -i predictorQuery.txt -o predictorOutput.txt --most_severe

The inbuilt annotation scheme attempts to produce similar results to VEP but it may not always be accurate. For example, it outputs the predicted amino acid substitution but the position of the amino acid is given simply based on the distance from the start of the coding sequence minus the intervening introns so it will be incorrect if some exons are not used.

--vep 'vep command'

The way that geneVarAssoc uses variant effect predictor (VEP) is to produce an input file for VEP called predictorQuery.txt, then run VEP to produce an output file called predictorOutput.txt containing the annotations, then read in that file. By default, the above command line will be run. The default value for the --vep argument is perl variant_effect_predictor.pl and to this geneVarAssoc appends -i predictorQuery.txt -o predictorOutput.txt --most_severe. If one needs a different command to run VEP successfully then the --vep argument is used to provide this. Because there will be spaces the whole argument must be enclosed in single quotes. For example, to run VEP under Unix the entry might read:

--vep 'perl /share/ref/VEP/ensembl-tools-release-87/scripts/variant_effect_predictor/variant_effect_predictor.pl --cache --dir /share/ref/VEP/Cache --merged --offline'

This would mean that the command that geneVarAssoc ran would be:

perl /share/ref/VEP/ensembl-tools-release-87/scripts/variant_effect_predictor/variant_effect_predictor.pl --cache --dir /share/ref/VEP/Cache --merged --offline -i predictorQuery.txt -o predictorOutput.txt --most_severe

To run VEP under Windows the entry might read:

--vep 'perl \vep87\vep.pl --cache --dir \cygwin64\home\dave\.vep --merged --offline'

If you have trouble getting this to work and geneVarAssoc reports an error saying that it could not read predictorOutput.txt then you should find that it has produced a file called predictorQuery.txt and can run VEP from the command prompt to find the command which you have to type before -i predictorQuery.txt -o predictorOutput.txt --most_severe in order to get it to run correctly.

--merge-alt-alleles 0 or 1

By default this is set to 1 and all alternate alleles are merged and treated as a single alternate allele. If instead --merge-alt-alleles 0 is used then they are treated separately as if there is a separate variant for each allele. So for a variant at 16:132368 two variants will be output, labelled 16:132368.1 and 16:132368.2, providing genotypes relevant to the first and second alternate alleles respectively. To illustrate this, the genotypes output to scoreassoc based on different genotypes in the vcf file are as shown below:

--merge-alt-alleles 1
0/0 1 1
0/1 1 2
0/2 1 2
1/2 2 2
2/2 2 2

--merge-alt-alleles 0
0/0 1 1 1 1
0/1 1 2 1 1
0/2 1 1 1 2
1/2 1 2 1 2
2/2 1 1 2 2

Breaking up the variant in this way allows the two alleles to be given their own annotations and treated separately and this will usually be a reasonable way to deal with multiallelic variants although it may be less appropriate if alternate alleles have a high frequency.

--consequence-threshold num

Tell scoreassoc to only use variants with an annotation number of num or higher (nonsynonymous=8 for the inbuilt scheme and protein_altering_variant=24 with VEP). See the section below on annotations for more details.

--use-consequence-weights 0 or 1

If 1, provide scoreassoc with functional weights based on the annotation for each variant, such that variants which are predicted to affect function are weighted more highly.

--only-use-SNPs 0 or 1

If 1, only use variants with two alleles, each consisting of a single base. Do not use indels.

--write-comments 0 or 1

If 1, provide information about each variant to scoreassoc to be output as comments. Default value is 1.

--write-score-file 0 or 1

If 1, instruct scoreassoc to output scores as well as standard output file.

--do-recessive-test 0 or 1

If 1, instruct scoreassoc to test for recessive effects.

--weight-threshold threshold

Instructs scoreassoc to only use those variants whose weight (based on rarity and function) exceeds the threshold.

--LD-threshold threshold

Instructs scoreassoc, when carrying out an analysis for recessive effects, to drop a variant which occurs in more than the stated proportion of subjects with another variant.

--add-chr 0 or 1 or 0011001, etc.

Some vcf files code give the chromosome entry for chromosome 17 as chr17, other as 17. If 1, adds chr to the chromosome number when running tabix to extract variants from all the vcf files. If 0, adds chr for none of them. If a sequence of 0's and 1's, equal in length to the number of vcf files, then adds chr to the vcf files corresponding to the 1's.

--GQ-threshold threshold

Only output genotypes reaching this threshold for the genotype quality score and treat others as unknown.

--depth-threshold threshold

Treat as unknown a genotype for which there are fewer reads than specified by the threshold.

--hetdev-threshold threshold

Treat as unknown a genotype which has been called as heterozygous but for which the numbers of reads for each allele (in the AD field) are very unequal. The expected number for each is 0.5 times the read depth (the total number) and the variance is 0.25 times the depth. The standard deviation is the square root of the variance. If the difference between the actual and expected number of reads exceeds the standard deviation multiplied by hetdev-threshold the genotype is treated as unknown. Thus one might specify --hetdev-threshold 3 to exclude heterozygous genotype calls which seem unlikely to be correct.

--AB-threshold threshold

Specify an allele balance threshold. A heterozygous genotype should have an approximately equal number of reads for each allele. If the proportion of either allele is below this threshold the genotype is treated as unknown, e.g. one might use –AB-threshold 0.2.

--unknown-if-untyped 0 or 1

If 1, treat a variant which has no entry in a vcf file as having unknown genotype for all subjects in that file. If 0, treat all subjects in that file as being wild type. If 1 and a frequency file is used then any variants missing from the frequency file will be omitted from the analysis. If 0 they will be treated as having frequency of 0 (i.e. all wild type). The assumption then is that the variant was adequately covered but did not appear in the vcf file because no alt alleles were seen. Default value is 0.

--wild-if-unknown 0 or 1

If 1, treat all unknown genotypes as wildtype. This can be useful if a number of vcf files have been merged and variants missing from one or more of the original files have been coded as unknown. However, doing this means one discards information about genotypes which really should be called as unknown because sufficient information was not available to call them reliably. Default value is 0.

--unknown-if-no-pass 0 or 1

If 1, variants without a PASS will be treated as unknown for that vcf file. If 0, the genotypes will be used. Default value is 0.

--skip-if-no-pass 0 or 1

If 1, variants without a PASS will be omitted rather than being treated as unknown. Default value is 0.

--ignore-alleles 0 or 1

If 1, treat variants at the same position in different vcf files as being the same variant and merge them, whether or not the REF and ALT alleles are the same in both. If 0, only treat them as the same variant if the REF and ALT allele strings are identical. Default value is 0.

--use-haplotypes 0 or 1

If 1, instruct scoreassoc to treat the genotypes as phased for recessive analyses. This is appropriate if the alleles are ordered within the genotypes or if trios are used. Default value is 0.

--show-hap-locus-names 0 or 1

If 1, instruct scoreassoc to output the names of the loci with minor alleles in subjects who are homozygotes or compound heterozygotes when carrying out recessive analyses based on haplotypes. Default value is 0.

--dottest 0 or 1

If 0, instruct scoreassoc not to perform a t test to compare scores between cases and controls.

--dolrtest 0 or 1

If 1, instruct scoreassoc to perform a likelihood ratio test based on logistic regression to compare scores between cases and controls.

--varfile filename

Provide scoreassoc with a file contain values of variables to be used in logistic regression analysis.

--testfile filename

Provide scoreassoc with a file to specify a likelihood ratio test using logistic regression analysis.

--start-from-fitted 0 or 1

When carrying out logistic regression analysis, use the fitted values obtained from the first model as the starting values to find the best fitting values for the second model.

--dont-extract-variants 0 or 1

If 1, skip over the step of extracting the variants from the vcf file(s) and assume that the necessary files have already been created locally. Used for debugging. Default value is 0.

--only-extract-variants 0 or 1

If 1, extracting the variants from the main vcf file(s) but do not go on to perform scoreassoc analyses. Temporary vcf files will be created and not deleted. Default value is 0.

--keep-temp-files 0 or 1

If 1, do not delete the temporary vcf files which are produced. This means one can examine them subsequently, for example to see which variants they contain, and can be useful for debugging. Default value is 0.

--do-not-run 0 or 1

If 1, set up all the files for the analysis but do not actually run scoreassoc. A bash script or, under MSDOS, a batch file will be created which can be run to perform the analysis. Can be useful for debugging. Default value is 0.

--debug 0 or 1

If 1, outputs intermediate values for expression evaluations and some other information which may be useful for debugging. Default value is 0.

--sequence-path pathname

This can be used to provide the name of the folder containing the main vcf files. If this is specified, then rather than providing the full path for each vcf file one would specify just the file name. This potentially allows different analyses to look for the same files in different places and is intended to facilitate using scratch space, etc.

--clear-cont 0 or 1

If 1, clears all the previously specified control files. This allows a subsequent arg-file to ignore what has preceded it and specify new control files from scratch.

--clear-case 0 or 1

If 1, clears all the previously specified case files. This allows a subsequent arg-file to ignore what has preceded it and specify new case files from scratch.

--exclusion-list filename or --

Reads exclusions to be used by scoreassoc one line at a time and passes them unchanged to scoreassoc. If instead of a filename -- is specified then the exclusions will be read from the current file until the end of file is reached.

--test-name name

By default, created files are named gva.GENE.*. This option replaces gva with the value name.

--weight-expression 'expression'

Provides an expression to use to provide weights for each locus instead of the default scheme based on the inbuilt or VEP annotations. For further information, see below in the sections on expressions and annotations.

--exclusion-expression 'expression'

Provides an expression which is evaluated for each variant and which if true (non-zero) will mean that the variant is marked not to be used. There can be any number of these expressions. These exclusions are applied by geneVarAssoc and are different from those in the exclusion list passed on to scoreAssoc. geneVarAssoc bases its exclusions on the property of the variant whereas scoreAssoc exclusions are based on allele and genotype counts. For further information, see below in the sections on expressions and annotations.

--comment-expression 'expression'

Provides an expression to use to enter into the comment for each locus which will be passed on to scoreAssoc, instead of the default information about the annotation used. For further information, see below in the sections on expressions and annotations.

 

Options to be used by other programs

Related programs use some of the same options as geneVarAssoc but also some specific to themselves. These programs will be documented subsequently.

--interval-list-file file

Specifies file containing list of intervals from which intVarAssoc will extract variables and perform a scoreassoc analysis. Intervals are in format chr:start-end, e.g. 5:140000-240000.

--position chr:pos

A coordinate is given consisting of a chromosome followed by a colon and then base position, to be used by related programs, not geneVarAssoc itself.

Expressions

An inbuilt parser is used to interpret the expressions which may be provided for the --weight-expression and --exclude-expression options. One can build up arbitrarily complex logical and arithmetic expressions and using the usual precedence rules and parentheses. The symbols &, | and = mean AND, OR and IS EQUAL TO. Values are either floating point numbers or strings and conversion between these happens automatically as required, so the number 3.7 converts into the string "3.7" and vice versa. If a string begins with a number then this will be its numerical value and otherwise its value will be 0. A logical expression has a numerical value of 1 if true and 0 if false and a numerical expression has a logical value of false if it is 0 and true if it is non-zero. Operators or functions can be unary or binary and the format for functions which take two arguments is the same as for operators taking two values in that the function name goes between the arguments, so that one can write, for example, 3+5 or "geneVar" STRCAT "Assoc".

A list of operators/functions is provided below and hopefully the example files will make it clearer how they can be used in practical situations.

GETWEIGHT

The first argument consists of an annotation, such as may be produced by the inbuilt functions or by variant effect predictor (VEP) but which can consist of any string. The second argument provides a weight table which assigns the weight for that annotation. The second argument may be "DEFAULTWEIGHTS" or "DEFAULTVEPWEIGHTS", in which case the default weights will be allocated for either an inbuilt or VEP annotation. Alternatively, the second argument can be the name of a file which provides the weights for each annotation, e.g. "MYWEIGHTS.txt". Each line of this file must contain an annotation string followed by the weight to be assigned for that annotation. For example, (ANNOT "INBUILT") GETWEIGHT "DEFAULTWEIGHTS" returns the default weight assigned for the annotation produced using the inbuilt annotation functions. (ANNOT "VEP") GETWEIGHT "MYWEIGHTS.txt" assigns a weight for the VEP annotation using the scheme defined in that file, so if the VEP annotation was "stop_gained" and the file contained a line saying stop_gained 25 then the expression would return a value of 25.

VCFLOOKUP

This function looks up the value of the information field entry denoted by the first argument in the VCF file specified by the second argument. For example, "AF" VCFLOOKUP "ExAC.r0.3.nonpsych.sites.vcf.gz" returns the value for AF (the allele frequency) in the ExAC VCF file. The function looks for a variant which is it at the same position as the variant currently under consideration. The VCF file may be an external file or it may be one of the files used to provide case or control genotypes or allele frequencies. So if the information field at the matching position in  VCF file contains the string …;AF=0.02;… then the returned result will be 0.02. If there is no string matching ;AF then the returned result will be e.g. "NOVCFENTRY_17_198232_AF" while if there is no line matching the required position it will return a result such as "NOVCFLINE_17_198232". The full path and filename of the VCF file must be provided. If it contains an asterisk then this will be replaced with the chromosome number. The VCF file must have been indexed with tabix.

VCFADDCHRLOOKUP

This function behaves in the same way as VCFLOOKUP except that it puts the letters CHR before the chromosome number when performing the lookup into the VCF file.

VCFADDLOWERCHRLOOKUP

This function behaves in the same way as VCFLOOKUP except that it puts the letters chr before the chromosome number when performing the lookup into the VCF file.

ANNOT

Argument should be "INBUILT" or "VEP". For the variant currently under consideration, it returns the string for the annotation produced by the inbuilt functions or by variant effect predictor (VEP). E.g. the result for the expression ANNOT "INBUILT" might be "INTRONIC"

ATTRIB

Returns a value describing an attribute of the variant currently under consideration. Argument should be one of the following: "WEIGHT" - produces the numerical value of the weight which has been assigned to the variant, e.g. 5; "POS" produces a string describing the base pair position, e.g. "1376826"; "CHR" produces a string describing the chromosome, e.g. "17"; "COORD" produces a string describing the coordinates,  e.g. "17:1376826"; "ID" produces a string consisting of the variant name, e.g. "rs785623"; "ISSNP" produces a value of 1 if the variant is an SNP, else 0.

GETPOLYPHEN

Takes one argument, a string, and looks within it for an entry corresponding to the POLYPHEN status output by VEP. Typically this would be applied to  the result of looking up the information field in a VCF file containing annotations produced by VEP. The function looks for the following strings: "probably_damaging", "possibly_damaging", "unknown", "benign" and if it finds one it returns the most severe. Thus if there are multiple entries for multiple transcripts only the most severe will be reported. If no entry is found it returns "polyphen_missing".

GETSIFT

As for GETPOLYPHEN except it looks for a SIFT status consisting of either "deleterious" or "tolerated", again reporting the most severe if there is more than one. If no entry is found returns "sift_missing".

GETVEP

As for GETPOLYPHEN except it looks for a VEP annotation and reports the most severe if there is more than one. If no entry is found returns "NULL_CONSEQUENCE". This feature allows a stored VEP annotation to be extracted from a VCF file containing annotation rather than using ANNOT to run VEP.

 

 

It is possible to construct quite complicated expressions whose results are then used by geneVarAssoc and when it is run normally one cannot see the intermediate stages of the expression evaluation. If one specifies --debug 1 then the intermediate values for each part of the expression are output as they are evaluated and this can be helpful for trouble-shooting.

If one wishes to allocate different weights from the default ones then the --weight-expression option can be used. The two internal tables of default weights are referred to as DEFAULTWEIGHTS and DEFAULTVEPWEIGHTS and if desired these can be used in a weight expression. This means that this:

--weight-expression '(ANNOT "INBUILT") GETWEIGHT "DEFAULTWEIGHTS"'

has the same effect as:

--use-ensembl 0

and:

--weight-expression '(ANNOT "VEP") GETWEIGHT "DEFAULTVEPWEIGHTS"'

has the same effect as:

--use-ensembl 1

(Note that the whole expression is enclosed in single quotes and the string values are enclosed in double quotes.)

To use different weights, one specifies an external file containing a weight table, each line of which contains and an annotation and the weight to be assigned to it. Example files containing the default weight schemes are provided as DEFAULTWEIGHTS.txt and VEPWEIGHTS.txt. To use different weights, one might copy one of these files to one called MYWEIGHTS.txt and then edit it to provide the desired weights for each annotation. Then one could specify, for example:

--weight-expression '(ANNOT "VEP") GETWEIGHT "MYWEIGHTS.txt"'

Thus, the user can arbitrarily assign any weight they choose to any of the annotation strings.

The same process can be applied to other forms of annotation. Suppose there is an external VCF file called annot.vcf which contains a SIFT annotation for each of the variants in the case/control VCF file(s). The SIFT annotation could be in the information field and could have a format such as …;SIFT=XX;… where XX is one of TOLERATED, DAMAGING or N/A. Then, one might have a file called SIFTWEIGHTS.txt with the following lines:

TOLERATED 5
DAMAGING 20
N/A 1

Then one could specify the following:

--weight-expression '("SIFT" VCFLOOKUP "annot.vcf") GETWEIGHT "SIFTWEIGHTS.txt"'

In practice, when one uses VEP to produce an annotated VCF file, all the annotations are stored together in an information field labelled CSQ. To extract the SIFT annotation from this one can use an expression such as:

--weight-expression 'GETSIFT("CSQ" VCFLOOKUP "annot.vcf") GETWEIGHT "SIFTWEIGHTS.txt"'

The GETSIFT function looks through the entire CSQ field and extracts the most severe SIFT annotation it can find.

Likewise, if one wish to extract a VEP annotation one could use an expression like:

--weight-expression 'GETVEP("CSQ" VCFLOOKUP "annot.vcf") GETWEIGHT "VEPWEIGHTS.txt"'

 

If one wishes to completely ignore variants having a particular annotation one can assign it a weight of 0. Such variants will have no effect on the scoreAssoc analysis. Another option is to use a suitable exclusion expression, e.g.:

--exclusion-expression (ANNOT "INBUILT") = "INTRONIC"

In order to ensure that weight expressions are being evaluated as intended, it can be helpful to specify --debug 1. Another tip is to temporarily specify --weight-factor 1. This means that scoreAssoc does not scale the weight according to rarity of the variant so that the weights which are output by scoreAssoc are the same as the functional weights which are provided by geneVarAssoc.

Exclusion expressions

Expressions can be provided to limit the variants to be used in the scoreAssoc analysis using different criteria. For example, if one wishes to restrict analysis to ultra-rare variants not seen in ExAC one might use:

--exclude-expression ' ("AF" VCFLOOKUP "ExAC.r0.3.nonpsych.sites.vcf.gz") > 0'

On the other hand, if one wishes to only include variants which have an entry in ExAC one could use:

--exclude-expression ' (("AF" VCFLOOKUP "ExAC.r0.3.nonpsych.sites.vcf.gz") STARTSWITH "NOVCFLINE"'

If one knows there is a poor quality variant at a particular position one might use:

--exclude-expression '(ATTRIB "COORD") = "3:2478686"'

Again, one should remember that these exclusions are based on properties of the variants and are different from those in the exclusion list which are passed on to  scoreAssoc, and which are based on allele and genotype counts.

Comment expressions

Expressions can be provided to define the text which will be output by scoreAssoc as the comment for each locus. By default these comments are based on the annotation which has been used. Any valid expression can be used, so for example one might use:

--comment-expression ' “MAF in ExAC is: “ STRCAT ("AF" VCFLOOKUP "ExAC.r0.3.nonpsych.sites.vcf.gz") '

This would provide the ExAC MAF as a comment in the final output file.

Or to include the complete VEP annotation as a comment one might use:

--comment-expression ' "CSQ" VCFLOOKUP "../vcf/SSS2/SSS2.vars.annot.vcf.gz"'

Please note that in the comments all spaces will be replaced by the underscore character.

Producing a file containing VEP annotations

The recommended way to use VEP is to first produce a file containing annotations for all loci and then to access it using the appropriate expression operators. This saves having to rerun VEP for every analysis.

In order to produce a VCF file containing VEP annotations for all variants contained in a case-control VCF file, one can proceed as follows.

Firstly, use the Unix cut utility to remove all but the first nine columns of the VCF file, discarding the genotypes which follow. Assuming the original file is called swe.vcf.gz, one can enter:

zcat swe.vcf.gz | cut -f1-9 > SSS2.vars.vcf

Then one can compress the resultant file:

bgzip SSS2.vars.vcf

Then to produce the annotated file using VEP one can enter a command such as:

perl /share/ref/VEP/ensembl-tools-release-87/scripts/variant_effect_predictor/variant_effect_predictor.pl --cache --dir /share/ref/VEP/Cache --merged --offline --input_file SSS2.vars.vcf.gz --format vcf --output_file SSS2.vars.annot.vcf --vcf --force_overwrite --sift b --polyphen b --pick

The --pick option has been specified to produce only one annotation, rather than one for every transcript. Otherwise genes with large numbers of transcripts can produce very long entries. Before it can be used the file containing the annotations must be compressed and indexed:

bgzip SSS2.vars.annot.vcf

tabix -p vcf SSS2.vars.annot.vcf.gz

When this process is completed the information field of annotation VCF file contains entries such as this:

CSQ=T|missense_variant|MODERATE|DRD2|ENSG00000149295|Transcript|ENST00000362072|protein_coding|2/8||||490|145|49|V/I|Gtc/Atc|||-1||HGNC|3023||tolerated(0.23)|possibly_damaging(0.556)

As shown in the example given below, the relevant information can be extracted using a combination of the VCFLOOKUP, GETSIFT and GETPOLYPHEN operators.

If --merge-alt-alleles 0 is used then the --pick option must not be used because one needs to have an annotation for every alternate allele. Even so, some alleles may not be matched correctly because of the way that VEP annotates indels.

Here is an example of a weight expression which uses the annotation file and builds up a weight from both the VEP annotation and from POLYPHEN and SIFT entries:

--weight-expression '( ( GETVEP ("CSQ" VCFLOOKUP "SSS2.vars.annot.vcf.gz")) GETWEIGHT "newWeights.200718.txt" ) + ( ( GETPOLYPHEN ("CSQ" VCFLOOKUP "SSS2.vars.annot.vcf.gz") ) GETWEIGHT "POLYPHENWEIGHTS.txt"  ) +  ( ( GETSIFT ("CSQ" VCFLOOKUP "SSS2.vars.annot.vcf.gz") ) GETWEIGHT "SIFTWEIGHTS.txt" ) '

Here is an example of a weight expression which uses CADD scores to assign weights of 0, 50, 150 or 300. It assumes that there are entries in the VCF file labelled CADD_phred=x where x is the CADD score:

--weight-expression '( ("CADD_phred" VCFLOOKUP "CADD-scores.vcf.gz")>0 IFTRUE 50) + ( ("CADD_phred" VCFLOOKUP "CADD-scores.vcf.gz")>10 IFTRUE 100) + ( ("CADD_phred" VCFLOOKUP "CADD-scores.vcf.gz")>30  IFTRUE 150)'

 

 

Trouble-shooting

Here are a few suggestions for things to try if the program does not seem to be working as expected.

Make sure that all file names and directories are correctly specified.

Check whether tabix works and produces the temporary vcf file (you can use --keep-temp-files). Copy the tabix command and enter it at the command prompt. Check that it produces a file and check that the file has variants in it, not just a header. Check whether you need to be adding chr to the chromosome number. Getting this wrong will mean that no variants can be extracted from the vcf file and you will get a message saying Failed to extract variants for this gene in the sao file. Another thing which can cause this behaviour is if you have --unknown-if-no-pass 1 and/or --skip-if-no-pass 1 but your vcf file does not actually have PASS in the relevant column.

To debug expressions, --debug 1 may help. Also, you can copy the expression or parts of it to be used as a comment expression so you can see what result it generates. Also, if you set --weight-factor 1 then scoreassoc will not change the variant weights according to frequency and will use the weights supplied by geneVarAssoc, which may make it easier to see what is happening with the weight expression.

Examine the input files produced for scoreassoc to see if they appear as expected. You can run the command from the .sh script.

You can make a debug version of geneVarAssoc by entering:

make -f geneVarAssoc.mak clean
make -f geneVarAssoc.mak DEBUGFLAG=-g

Then you can run it in gdb, which lets you step through the code, examine variables etc.
